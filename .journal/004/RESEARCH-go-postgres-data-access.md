# PostgreSQL Data-Access Layer Choices for a Hexagonal Go API Template (2025–2026)

## Executive summary

The Go community's recommendation for PostgreSQL data access has measurably shifted in the last ~12 months: the center of gravity has moved away from full-magic ORMs (GORM) and toward **sqlc** — generating type-safe Go code from hand-written SQL — as the default for production services that value performance, compile-time guarantees, and explicit query control ([dev.to](https://dev.to/bitsofmandal-yt/you-dont-need-gorm-there-is-a-better-alternative-12j2), [glukhov.org](https://www.glukhov.org/post/2025/09/comparing-go-orms-gorm-ent-bun-sqlc/), [reintech.io](https://reintech.io/blog/sqlc-vs-gorm-vs-sqlx-go-database-libraries-compared-2026)). The perception of "mixed opinions" is accurate but nuanced: the consensus is **not** that ORMs are universally bad — it is that tool choice is now framed conditionally by team, query profile, and performance needs, with sqlc as the strong default and ORMs reserved for CRUD-heavy, association-rich, rapid-development scenarios ([leapcell.io](https://leapcell.io/blog/sqlc-vs-gorm-two-approaches-to-database-interaction-in-go), [reintech.io](https://reintech.io/blog/sqlc-vs-gorm-vs-sqlx-go-database-libraries-compared-2026), [encore.dev](https://encore.dev/resources/go-orms)). A common emerging pattern is **combining** tools: sqlc for static type-safe queries, a query builder (Squirrel) for dynamic queries, and a standalone migration tool (Goose/golang-migrate/Atlas) ([bytebase.com](https://www.bytebase.com/blog/golang-orm-query-builder/)). For a hexagonal template whose persistence adapter sits behind a consumer-defined repository port and is tested with testcontainers, **sqlc + pgx/v5 + a standalone migration tool**, validated with the Testcontainers Postgres module's snapshot/restore feature, is the best-fit recommendation.

## Findings

### Theme 1: sqlc provides compile-time-validated type safety with near-raw-SQL performance

**Confidence: high** (multiple sources incl. primary docs; unanimous votes on the core claims)

sqlc parses your hand-written SQL and DDL using the real PostgreSQL parser, then generates Go code with exact struct types; if a query or schema input changes and the code isn't updated, it fails to compile rather than failing in production ([leapcell.io](https://leapcell.io/blog/sqlc-vs-gorm-two-approaches-to-database-interaction-in-go), [dev.to](https://dev.to/bitsofmandal-yt/you-dont-need-gorm-there-is-a-better-alternative-12j2), [medium.com](https://medium.com/@yashbatra11111/go-postgres-with-sqlc-the-zero-orm-stack-cloudflare-uses-for-99-99-uptime-d4beaddbfdf8)). The validation is effectively two gates — `sqlc generate` rejects invalid SQL at generation time, and the regenerated Go is then checked by the Go compiler — both of which precede runtime ([dev.to](https://dev.to/bitsofmandal-yt/you-dont-need-gorm-there-is-a-better-alternative-12j2), [reintech.io](https://reintech.io/blog/sqlc-vs-gorm-vs-sqlx-go-database-libraries-compared-2026)).

Because the generated code is a thin wrapper over `database/sql` or pgx with no runtime reflection, sqlc delivers essentially raw-SQL performance ([glukhov.org](https://www.glukhov.org/post/2025/09/comparing-go-orms-gorm-ent-bun-sqlc/), [reintech.io](https://reintech.io/blog/sqlc-vs-gorm-vs-sqlx-go-database-libraries-compared-2026)). It also has minimal runtime dependencies and integrates cleanly with external migration tools (golang-migrate, goose, atlas, dbmate, tern) and CI, making it the recommended choice for teams comfortable with SQL who want maximum control and speed ([glukhov.org](https://www.glukhov.org/post/2025/09/comparing-go-orms-gorm-ent-bun-sqlc/)). sqlc is consistently positioned as an ORM alternative for developers who prefer to write SQL ([encore.dev](https://encore.dev/resources/go-orms)).

### Theme 2: GORM trades compile-time safety and performance for convenience

**Confidence: high** (multiple current sources, including critical and neutral benchmarks; unanimous votes)

GORM relies on runtime reflection to map columns to struct fields, which yields only medium, runtime-level type safety: mistyped column names or invalid queries can reach production without compile-time validation, since arguments pass as `interface{}` ([glukhov.org](https://www.glukhov.org/post/2025/09/comparing-go-orms-gorm-ent-bun-sqlc/), [dev.to](https://dev.to/bitsofmandal-yt/you-dont-need-gorm-there-is-a-better-alternative-12j2), [leapcell.io](https://leapcell.io/blog/sqlc-vs-gorm-two-approaches-to-database-interaction-in-go)). That reflection adds overhead — often **negligible for typical CRUD** but measurable at scale and on hot paths — and GORM is also prone to N+1 query problems unless eager loading (Joins/selective Preload) is used carefully ([glukhov.org](https://www.glukhov.org/post/2025/09/comparing-go-orms-gorm-ent-bun-sqlc/), [leapcell.io](https://leapcell.io/blog/sqlc-vs-gorm-two-approaches-to-database-interaction-in-go), [reintech.io](https://reintech.io/blog/sqlc-vs-gorm-vs-sqlx-go-database-libraries-compared-2026)).

Importantly, the performance gap is not unconditional: for small result sets the database round-trip dominates and the difference is immaterial; the gap only becomes significant at large row counts ([leapcell.io](https://leapcell.io/blog/sqlc-vs-gorm-two-approaches-to-database-interaction-in-go), [reintech.io](https://reintech.io/blog/sqlc-vs-gorm-vs-sqlx-go-database-libraries-compared-2026)). GORM remains a defensible pick where rapid development, convenience, and a higher level of abstraction over Go structs matter more than raw performance — particularly CRUD-heavy, association-rich applications ([leapcell.io](https://leapcell.io/blog/sqlc-vs-gorm-two-approaches-to-database-interaction-in-go), [reintech.io](https://reintech.io/blog/sqlc-vs-gorm-vs-sqlx-go-database-libraries-compared-2026)).

### Theme 3: The recommendation is conditional, not a single winner — and ORMs are advised against for performance-critical / control-sensitive systems

**Confidence: high** (multiple independent 2025–2026 sources converge; unanimous votes)

Current sources frame the decision as "it depends" rather than crowning a universal best tool. Encore advises caution about ORMs in performance-critical systems or where direct control over the database is needed, because they add overhead and can obscure what happens at the database level ([encore.dev](https://encore.dev/resources/go-orms)). The Leapcell tradeoff framing is representative: choose sqlc for full SQL control, performance, and compile-time guarantees in a SQL-first workflow; choose GORM for rapid development, convenience, and struct-based abstraction ([leapcell.io](https://leapcell.io/blog/sqlc-vs-gorm-two-approaches-to-database-interaction-in-go)). Reintech extends this to three tools: sqlc for performance/compile-time safety/db-specific features, GORM for development speed and CRUD-heavy association-rich apps, and sqlx for minimal abstraction when the team is comfortable writing SQL — explicitly noting all three power production systems and that architecture/team/perf needs matter more than picking "the best" ([reintech.io](https://reintech.io/blog/sqlc-vs-gorm-vs-sqlx-go-database-libraries-compared-2026)).

### Theme 4: Hybrid stacks are an emerging recommended pattern

**Confidence: medium** (vendor/blog sources, but the pattern is independently corroborated as mainstream)

Rather than picking one approach, recent guidance recommends combining tools by team and query profile. Bytebase recommends, for teams with SQL expertise needing type safety across a mix of static and dynamic queries, the combination **sqlc + Squirrel + Goose** — sqlc for type-safe static queries, Squirrel for dynamic query construction, and Goose for migrations ([bytebase.com](https://www.bytebase.com/blog/golang-orm-query-builder/)). This directly addresses sqlc's main weakness — limited dynamic-query support — by pairing it with a query builder. A specific concrete production stack advocated as superior to GORM is **sqlc + pgx/v5 + Atlas**, citing less boilerplate, compile-time type safety, idiomatic Go, ~30–50% better performance, and explicit SQL control (while conceding GORM suits prototypes and internal tools) ([dev.to](https://dev.to/bitsofmandal-yt/you-dont-need-gorm-there-is-a-better-alternative-12j2)).

### Theme 5: Query builders occupy a distinct middle category, and Bob is a notable progressive-adoption hybrid

**Confidence: high** (primary self-positioning docs; unanimous votes)

Query builders are a separate category from ORMs. **goqu** explicitly positions itself as an expressive SQL builder and executor and states it is *not* intended as an ORM (redirecting users wanting associations/hooks to GORM); its core design goal is an expressive DSL that catches common SQL errors at compile time — a partial type-safety guarantee that is the main differentiator query builders offer over raw string SQL, though weaker than sqlc's full schema-validated type safety ([goqu docs](http://doug-martin.github.io/goqu/)).

**Bob** is a notable newer entrant that spans the full spectrum: it is a unified SQL toolkit functioning as both a query builder and a code-generating ORM/factory generator, progressively adoptable from raw SQL strings up to fully typed generated queries, models, and factories ([github.com/stephenafamo/bob](https://github.com/stephenafamo/bob)). Its four layers map directly onto the competing approaches: (1) a Squirrel-like query builder, (2) SQLBoiler-like ORM generation from schema, (3) FactoryBot-style factory generation for testing, and (4) sqlc-like SQL query code generation ([github.com/stephenafamo/bob](https://github.com/stephenafamo/bob)). Notably, Bob explicitly positions itself *against* traditional ORM "magic" — "convenience, not magic… no unexplainable magic, or needless abstraction" — reflecting the same community sentiment that has eroded heavyweight-ORM dominance ([github.com/stephenafamo/bob](https://github.com/stephenafamo/bob)).

### Theme 6: ent is the strongest typed-ORM option for large, refactor-heavy teams

**Confidence: medium** (current sources incl. neutral CI benchmark; one split vote on relative speed)

For teams that want ORM ergonomics *with* compile-time safety, ent provides high compile-time type safety via generated code that avoids runtime reflection, making it well-suited to large codebases, complex schemas, and refactor-heavy teams with strict typing needs ([glukhov.org](https://www.glukhov.org/post/2025/09/comparing-go-orms-gorm-ent-bun-sqlc/)). A neutral CI-generated benchmark shows ent beating GORM on insert, update, and read, supporting the (hedged) claim that ent is often faster than heavy runtime-magic ORMs — though a single self-run blog benchmark disputes the read/execution-time ranking, so the "often" qualifier matters ([glukhov.org](https://www.glukhov.org/post/2025/09/comparing-go-orms-gorm-ent-bun-sqlc/)).

### Theme 7: Testcontainers makes the hexagonal repository adapter testable against real Postgres, fast

**Confidence: high** (primary testcontainers-go docs; unanimous votes)

The Testcontainers for Go Postgres module runs a real PostgreSQL database inside a Docker container for tests, started via `Run(ctx, img, opts...)`, which replaced the deprecated `RunContainer()` as of v0.32.0 ([golang.testcontainers.org](https://golang.testcontainers.org/modules/postgres/)). Its snapshot/restore feature lets each test obtain a clean database by calling `Restore()` in cleanup — without recreating the container per test — enabling fast, isolated repository tests (create a snapshot after migrations, then restore between tests) ([golang.testcontainers.org](https://golang.testcontainers.org/modules/postgres/)). (Minor version caveat: the Snapshot/Restore methods themselves predate v0.32.0; v0.32.0 added a faster SQL-driver path. Known constraints include not naming the DB "postgres" and care with non-default usernames.) This testing approach is agnostic to the data-access library chosen, so it works equally well behind a consumer-defined repository port regardless of whether the adapter uses sqlc, pgx, or an ORM.

## Recommendation

**For the hexagonal Go API template: use sqlc + pgx/v5 as the persistence adapter behind the consumer-defined repository port, paired with a standalone migration tool (Goose, golang-migrate, or Atlas) and tested with the Testcontainers Postgres module.**

Rationale tied to the template's specific constraints:

- **Hexagonal fit.** sqlc generates plain functions over a generic DB connection interface with no runtime layer ([glukhov.org](https://www.glukhov.org/post/2025/09/comparing-go-orms-gorm-ent-bun-sqlc/)). The generated query code is a clean implementation detail that lives entirely inside the persistence adapter; the consumer-defined repository port can wrap generated methods without leaking ORM types, models, or "magic" into the domain. This is the cleanest fit for ports & adapters, and aligns with the anti-magic / explicit-SQL sentiment now dominant ([encore.dev](https://encore.dev/resources/go-orms), [github.com/stephenafamo/bob](https://github.com/stephenafamo/bob)).
- **Compile-time guarantees.** Schema/query changes break the build, not production ([dev.to](https://dev.to/bitsofmandal-yt/you-dont-need-gorm-there-is-a-better-alternative-12j2), [leapcell.io](https://leapcell.io/blog/sqlc-vs-gorm-two-approaches-to-database-interaction-in-go)) — valuable in a template meant to be copied and extended by many teams.
- **Performance & query control.** Near-raw-SQL performance with no reflection, full control of joins/indices/query plans, and no hidden N+1 behavior ([glukhov.org](https://www.glukhov.org/post/2025/09/comparing-go-orms-gorm-ent-bun-sqlc/), [reintech.io](https://reintech.io/blog/sqlc-vs-gorm-vs-sqlx-go-database-libraries-compared-2026)).
- **Testability.** The adapter is verified against real Postgres via Testcontainers with snapshot/restore for fast isolated tests ([golang.testcontainers.org](https://golang.testcontainers.org/modules/postgres/)).
- **Migrations stay external.** sqlc integrates with golang-migrate/goose/atlas and CI rather than owning migrations ([glukhov.org](https://www.glukhov.org/post/2025/09/comparing-go-orms-gorm-ent-bun-sqlc/)), keeping the template modular.

**Add a query builder (Squirrel) only if/when dynamic queries are needed**, following the widely recommended sqlc-for-static / builder-for-dynamic / standalone-migrations composition ([bytebase.com](https://www.bytebase.com/blog/golang-orm-query-builder/)). For a clean template, keep Squirrel optional to avoid premature dependency.

**When NOT to use this default:** if the target users are building CRUD-heavy, association-rich apps prioritizing development speed over performance, GORM is defensible ([leapcell.io](https://leapcell.io/blog/sqlc-vs-gorm-two-approaches-to-database-interaction-in-go), [reintech.io](https://reintech.io/blog/sqlc-vs-gorm-vs-sqlx-go-database-libraries-compared-2026)); if a large team wants ORM ergonomics with compile-time typing, ent is the better ORM ([glukhov.org](https://www.glukhov.org/post/2025/09/comparing-go-orms-gorm-ent-bun-sqlc/)). Because the chosen design hides persistence behind a repository port, swapping the adapter later is low-cost — reinforcing sqlc + pgx as a safe default.

## Caveats

- **Source quality is uneven.** Several supporting sources are vendor or blog content: Encore ([encore.dev](https://encore.dev/resources/go-orms)) and Bytebase ([bytebase.com](https://www.bytebase.com/blog/golang-orm-query-builder/)) are vendor blogs; reintech.io is an SEO blog with an explicitly AI-generated byline and no published benchmark numbers ([reintech.io](https://reintech.io/blog/sqlc-vs-gorm-vs-sqlx-go-database-libraries-compared-2026)). The strongest claims (sqlc's mechanism, type safety, reflection-free codegen; testcontainers API) are backed by primary docs or sqlc's own author; the comparative/recommendation claims rest on corroborated-but-secondary material.
- **The performance gap is frequently overstated.** Multiple sources stress that for typical workloads the database round-trip dominates and the sqlc-vs-GORM difference is immaterial — GORM can even be faster on small result sets; the gap only matters at large row counts or hot paths ([leapcell.io](https://leapcell.io/blog/sqlc-vs-gorm-two-approaches-to-database-interaction-in-go), [reintech.io](https://reintech.io/blog/sqlc-vs-gorm-vs-sqlx-go-database-libraries-compared-2026)). Performance alone is a weaker justification than compile-time safety and query control.
- **One refuted claim.** A Bytebase assertion that sqlc offers the best type safety *and performance for complex queries* while GORM incurs reflection overhead failed verification (1-2 vote) and is excluded — sqlc's weakness with complex/dynamic queries is real.
- **Time-sensitivity and tool churn.** GORM v1.30+ (mid-2025) added an opt-in generics-based API and codegen tool for improved type safety, narrowing the type-safety gap for teams who adopt the new API ([dev.to](https://dev.to/bitsofmandal-yt/you-dont-need-gorm-there-is-a-better-alternative-12j2)); the default/dominant GORM path still uses reflection. Bob is actively evolving and could mature into a stronger progressive-adoption option. A few cited articles (e.g., Bytebase, Apr 2025; goqu's stable self-description) sit at or slightly outside a strict 12-month window but remain corroborated by current sources.
- **Version detail on testcontainers** Snapshot/Restore predates v0.32.0; the claim's "introduced v0.32.0" conflates the feature with the v0.32.0 SQL-driver speedup ([golang.testcontainers.org](https://golang.testcontainers.org/modules/postgres/)).

## Open questions

1. **Dynamic-query strategy.** sqlc handles static SQL well but is weak on dynamic queries. Should the template ship with Squirrel (or Bob's query-builder layer) integrated from the start, or leave it as documented guidance — and what's the cleanest way to expose dynamic queries through a repository port without leaking the builder?
2. **Migration tool choice.** Sources name golang-migrate, goose, and atlas interchangeably as sqlc-compatible; which best fits a template's CI and testcontainers snapshot-after-migration workflow was not resolved by the evidence.
3. **Where does Bob fit for a template?** Bob's progressive-adoption model is conceptually attractive for a starter template, but no benchmark, adoption-scale, or production-maturity evidence comparable to sqlc's was surfaced — is it production-ready enough to recommend over plain sqlc?
4. **pgx vs database/sql for the generated code.** sqlc supports both; the sources favor pgx/v5 for PostgreSQL but did not provide head-to-head evidence (connection pooling, COPY, LISTEN/NOTIFY, performance) to firmly justify pgx over `database/sql` within a hexagonal adapter.
