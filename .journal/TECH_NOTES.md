# Technical Notes

- Use hexagonal architecture at all times. Keep business logic isolated from CLI, filesystem, network, storage, and other external adapters.
- Prefer functional testing before calling any feature complete. Unit tests are useful, but they do not prove the tool works the way the design intends.
- Take an agile approach to development. Avoid waterfall: underspecify when useful, prototype early, learn from the result, and refine from working behavior.

The code + `README.md` are the source of truth for architecture and usage; `.journal/TARGET_SHAPE.md` is the original design doc, now largely realized (historical). - **`internal/` is structured per-domain** (PR #8 `1f1e5a7`). Each domain owns its adapters nested beneath its core package â€” e.g. `internal/todo` (core) + `internal/todo/{httpapi,postgres}`. Shared, cross-domain infrastructure stays under `internal/adapter/`: the generic chi/Huma transport `internal/adapter/http` (+ `middleware`, `problem`) and the Postgres pool/`Connect` + goose migrate library + embedded `migrations/` under `internal/adapter/postgres` (schema/migrations are DB-level, not per-domain). sqlc `out`/`queries` live with the domain (`internal/todo/postgres/{sqlc,queries}`); sqlc `schema` still points at the shared `internal/adapter/postgres/migrations`. Add a resource by mirroring `internal/todo/{httpapi,postgres}` and registering it in `app.go` â€” alias the like-named per-domain `httpapi`/`postgres` imports (see `todopostgres` in `app.go`).
- Convention: after changing the API, run `moon run openapi` to refresh the committed `docs/docs/openapi.yaml`; CI (`root:check` â†’ `openapi-check`) fails if it drifts.
- **PostgreSQL persistence tier is built** (PR #6 `18b56e7`) and is now the **only** tier (PR #9 `8a46286` â€” see the PostgreSQL-only bullet). The code + `README.md` "Persistence" section are the source of truth, design doc at `.journal/004/POSTGRES_TIER.md`. Stack: jackc/pgx v5 + **sqlc** (generated code committed under `internal/todo/postgres/sqlc/`, drift-guarded by `root:sqlc-check`) + **goose** migrations (embedded; explicit `migrate up|down|status` subcommand, NEVER auto-run on serve). sqlc reads the goose migrations as its schema (single source). No query-builder shipped â€” dynamic queries via `sqlc.narg`, with Squirrel/Bob documented as adapter-internal escape hatches. `Save` is full insert-or-replace (created_at included). `app.New(ctx, cfg, logger, version, ...Option)` (pool lifecycle + postgres `/readyz`); the `app.WithRepository(todo.Repository)` option injects a repo and skips the DB connect â€” the seam for no-DB wiring tests and for integrators adding a custom adapter without editing `resolveStore`.
- **Day-one Docker Compose stack is built** (PR #7 `8b68bd4`). `docker compose up --build` runs an ordered DAG â€” `postgres` (healthcheck) â†’ `migrate` (one-shot `migrate up`) â†’ `seed` (one-shot) â†’ `api` (`serve`, postgres via `TEMPLATE_GO_API_DATABASE_URL`) â€” against an **ephemeral** Postgres (no persisted volume, so every `up` is a clean, migrated, seeded DB). Seeds are drop-in `*.sql` under `hack/sql/`, applied by a `postgres:17-alpine` psql one-shot **after** migrations (NOT the Postgres init-dir, which runs pre-migration and would fail on `INSERT`). Reuses the existing root `Dockerfile` unchanged (`migrate`/`api` share one built image via a compose anchor); `compose.yaml` lives at repo root; README "Local stack (Docker Compose)" documents it. Design doc/postmortem: `.journal/006/SUMMARY.md`.
- Conventions added: CLIs are pinned via **Proto** (`.moon/proto/*.toml` + `.prototools`), NOT `go tool`. Integration tests live in **`internal/integration`** (package `integration`, `//go:build integration`, a non-tagged `doc.go` anchor), run via `moon run test-integration` (needs Docker); unit tests stay beside the code. golangci lints tagged files via `run.build-tags`. In moon tasks use `-tags integration` (space, not `=`). After changing migrations/queries run `moon run sqlc` and commit. Test doubles use **mockery** (Proto-pinned `mockery`, `.mockery.yaml` testify template, generated mock committed at `internal/todo/mocks`, drift-guarded by `root:mockery-check` â€” mirrors the sqlc generate/check pattern); after changing a mocked port run `moon run mockery` and commit. Repository doubles: the generated `mocks.Repository` for interaction/error-injection unit tests (`service_test.go`), a stateful in-memory fake `internal/todo/todotest` for end-to-end HTTP tests + no-DB app wiring. mockery release assets are `mockery_{version}_{Darwin|Linux}_{x86_64|arm64}.tar.gz` (note `x86_64`, not `amd64`).
