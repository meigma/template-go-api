# Technical Notes

- Use hexagonal architecture at all times. Keep business logic isolated from CLI, filesystem, network, storage, and other external adapters.
- Prefer functional testing before calling any feature complete. Unit tests are useful, but they do not prove the tool works the way the design intends.
- Take an agile approach to development. Avoid waterfall: underspecify when useful, prototype early, learn from the result, and refine from working behavior.

- **Day-one Docker Compose stack is built** (PR #7 `8b68bd4`). `docker compose up --build` runs an ordered DAG â€” `postgres` (healthcheck) â†’ `migrate` (one-shot `migrate up`) â†’ `seed` (one-shot) â†’ `api` (`serve --store=postgres`) â€” against an **ephemeral** Postgres (no persisted volume, so every `up` is a clean, migrated, seeded DB). Seeds are drop-in `*.sql` under `hack/sql/`, applied by a `postgres:17-alpine` psql one-shot **after** migrations (NOT the Postgres init-dir, which runs pre-migration and would fail on `INSERT`). Reuses the existing root `Dockerfile` unchanged (`migrate`/`api` share one built image via a compose anchor); `compose.yaml` lives at repo root; README "Local stack (Docker Compose)" documents it. Design doc/postmortem: `.journal/006/SUMMARY.md`.
