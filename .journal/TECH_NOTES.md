# Technical Notes

- Use hexagonal architecture at all times. Keep business logic isolated from CLI, filesystem, network, storage, and other external adapters.
- Prefer functional testing before calling any feature complete. Unit tests are useful, but they do not prove the tool works the way the design intends.
- Take an agile approach to development. Avoid waterfall: underspecify when useful, prototype early, learn from the result, and refine from working behavior.
- The API-server template is **built** (slices 1–2 merged: PR #4 `745a9ed`, PR #5 `05f5446`). The code + `README.md` are the source of truth for architecture and usage; `.journal/TARGET_SHAPE.md` is the original design doc, now largely realized (historical). Shape as built: chi v5 + Huma v2 (transport-scoped, code-first OpenAPI 3.0.3); pragmatic ports & adapters (per-domain layout — see the dedicated bullet below); domain core `internal/todo` with nested `internal/todo/{httpapi,memory,postgres}`, shared infra under `internal/adapter/{http,postgres}`, plus `internal/{config,observability,logctx,app,cli,integration}`; slog + Prometheus `/metrics` on a **dedicated listener** (`--metrics-addr`, default `:9090`); RFC 9457 on every non-Huma surface; OpenAPI exported server-less → neoteroi OAD render with a `root:check` drift-guard.
- **`internal/` is structured per-domain** (PR #8 `1f1e5a7`). Each domain owns its adapters nested beneath its core package — e.g. `internal/todo` (core) + `internal/todo/{httpapi,memory,postgres}`. Shared, cross-domain infrastructure stays under `internal/adapter/`: the generic chi/Huma transport `internal/adapter/http` (+ `middleware`, `problem`) and the Postgres pool/`Connect` + goose migrate library + embedded `migrations/` under `internal/adapter/postgres` (schema/migrations are DB-level, not per-domain). sqlc `out`/`queries` live with the domain (`internal/todo/postgres/{sqlc,queries}`); sqlc `schema` still points at the shared `internal/adapter/postgres/migrations`. Add a resource by mirroring `internal/todo/{httpapi,memory,postgres}` and registering it in `app.go` — alias the like-named per-domain `httpapi`/`postgres` imports (see `todopostgres` in `app.go`).
- Convention: after changing the API, run `moon run openapi` to refresh the committed `docs/docs/openapi.yaml`; CI (`root:check` → `openapi-check`) fails if it drifts.
- **PostgreSQL persistence tier is built** (PR #6 `18b56e7`). `--store=memory|postgres` selects the adapter (memory is the zero-infra default); the code + `README.md` "Persistence" section are the source of truth, design doc at `.journal/004/POSTGRES_TIER.md`. Stack: jackc/pgx v5 + **sqlc** (generated code committed under `internal/todo/postgres/sqlc/`, drift-guarded by `root:sqlc-check`) + **goose** migrations (embedded; explicit `migrate up|down|status` subcommand, NEVER auto-run on serve). sqlc reads the goose migrations as its schema (single source). No query-builder shipped — dynamic queries via `sqlc.narg`, with Squirrel/Bob documented as adapter-internal escape hatches. `Save` is full insert-or-replace (created_at included), identical to the memory adapter. `app.New` is `New(ctx) (*App, error)` (pool lifecycle + postgres `/readyz`).
- **Day-one Docker Compose stack is built** (PR #7 `8b68bd4`). `docker compose up --build` runs an ordered DAG — `postgres` (healthcheck) → `migrate` (one-shot `migrate up`) → `seed` (one-shot) → `api` (`serve --store=postgres`) — against an **ephemeral** Postgres (no persisted volume, so every `up` is a clean, migrated, seeded DB). Seeds are drop-in `*.sql` under `hack/sql/`, applied by a `postgres:17-alpine` psql one-shot **after** migrations (NOT the Postgres init-dir, which runs pre-migration and would fail on `INSERT`). Reuses the existing root `Dockerfile` unchanged (`migrate`/`api` share one built image via a compose anchor); `compose.yaml` lives at repo root; README "Local stack (Docker Compose)" documents it. Design doc/postmortem: `.journal/006/SUMMARY.md`.
- Conventions added: CLIs are pinned via **Proto** (`.moon/proto/*.toml` + `.prototools`), NOT `go tool`. Integration tests live in **`internal/integration`** (package `integration`, `//go:build integration`, a non-tagged `doc.go` anchor), run via `moon run test-integration` (needs Docker); unit tests stay beside the code. golangci lints tagged files via `run.build-tags`. In moon tasks use `-tags integration` (space, not `=`). After changing migrations/queries run `moon run sqlc` and commit.
- **Planned direction (not yet done):** remove the in-memory tier and default the template to PostgreSQL-only — the `memory` adapter and the `--store` toggle are intended to be dropped to cut boilerplate consumers would delete. The compose stack already assumes postgres (sets `--store=postgres` explicitly). Until then, `memory` remains the zero-infra default.
- Future-slice seams left open (not yet built): authn/authz; OTel tracing; rate limiting; pagination; API versioning; mockery. Near-term follow-up: wire `test-integration` into CI once the `.disabled` GitHub workflows get a Docker-capable runner.
