# Technical Notes

- Use hexagonal architecture at all times. Keep business logic isolated from CLI, filesystem, network, storage, and other external adapters.
- Prefer functional testing before calling any feature complete. Unit tests are useful, but they do not prove the tool works the way the design intends.
- Take an agile approach to development. Avoid waterfall: underspecify when useful, prototype early, learn from the result, and refine from working behavior.
- The API-server template is **built** (slices 1–2 merged: PR #4 `745a9ed`, PR #5 `05f5446`). The code + `README.md` are the source of truth for architecture and usage; `.journal/TARGET_SHAPE.md` is the original design doc, now largely realized (historical). Shape as built: chi v5 + Huma v2 (transport-scoped, code-first OpenAPI 3.0.3); pragmatic ports & adapters (per-domain layout — see the dedicated bullet below); domain core `internal/todo` with nested `internal/todo/{httpapi,postgres}` (+ test-support `mocks`/`todotest`), shared infra under `internal/adapter/{http,postgres}`, plus `internal/{config,observability,logctx,app,cli,integration}`; slog + Prometheus `/metrics` on a **dedicated listener** (`--metrics-addr`, default `:9090`); RFC 9457 on every non-Huma surface; OpenAPI exported server-less → neoteroi OAD render with a `root:check` drift-guard.
- **`internal/` is structured per-domain** (PR #8 `1f1e5a7`). Each domain owns its adapters nested beneath its core package — e.g. `internal/todo` (core) + `internal/todo/{httpapi,postgres}`. Shared, cross-domain infrastructure stays under `internal/adapter/`: the generic chi/Huma transport `internal/adapter/http` (+ `middleware`, `problem`) and the Postgres pool/`Connect` + goose migrate library + embedded `migrations/` under `internal/adapter/postgres` (schema/migrations are DB-level, not per-domain). sqlc `out`/`queries` live with the domain (`internal/todo/postgres/{sqlc,queries}`); sqlc `schema` still points at the shared `internal/adapter/postgres/migrations`. Add a resource by mirroring `internal/todo/{httpapi,postgres}` and registering it in `app.go` — alias the like-named per-domain `httpapi`/`postgres` imports (see `todopostgres` in `app.go`).
- Convention: after changing the API, run `moon run openapi` to refresh the committed `docs/docs/openapi.yaml`; CI (`root:check` → `openapi-check`) fails if it drifts.
- **PostgreSQL persistence tier is built** (PR #6 `18b56e7`) and is now the **only** tier (PR #9 `8a46286` — see the PostgreSQL-only bullet). The code + `README.md` "Persistence" section are the source of truth, design doc at `.journal/004/POSTGRES_TIER.md`. Stack: jackc/pgx v5 + **sqlc** (generated code committed under `internal/todo/postgres/sqlc/`, drift-guarded by `root:sqlc-check`) + **goose** migrations (embedded; explicit `migrate up|down|status` subcommand, NEVER auto-run on serve). sqlc reads the goose migrations as its schema (single source). No query-builder shipped — dynamic queries via `sqlc.narg`, with Squirrel/Bob documented as adapter-internal escape hatches. `Save` is full insert-or-replace (created_at included). `app.New(ctx, cfg, logger, version, ...Option)` (pool lifecycle + postgres `/readyz`); the `app.WithRepository(todo.Repository)` option injects a repo and skips the DB connect — the seam for no-DB wiring tests and for integrators adding a custom adapter without editing `resolveStore`.
- **Day-one Docker Compose stack is built** (PR #7 `8b68bd4`). `docker compose up --build` runs an ordered DAG — `postgres` (healthcheck) → `migrate` (one-shot `migrate up`) → `seed` (one-shot) → `api` (`serve`, postgres via `TEMPLATE_GO_API_DATABASE_URL`) — against an **ephemeral** Postgres (no persisted volume, so every `up` is a clean, migrated, seeded DB). Seeds are drop-in `*.sql` under `hack/sql/`, applied by a `postgres:17-alpine` psql one-shot **after** migrations (NOT the Postgres init-dir, which runs pre-migration and would fail on `INSERT`). Reuses the existing root `Dockerfile` unchanged (`migrate`/`api` share one built image via a compose anchor); `compose.yaml` lives at repo root; README "Local stack (Docker Compose)" documents it. Design doc/postmortem: `.journal/006/SUMMARY.md`.
- Conventions added: CLIs are pinned via **Proto** (`.moon/proto/*.toml` + `.prototools`), NOT `go tool`. Integration tests live in **`internal/integration`** (package `integration`, `//go:build integration`, a non-tagged `doc.go` anchor), run via `moon run test-integration` (needs Docker); unit tests stay beside the code. golangci lints tagged files via `run.build-tags`. In moon tasks use `-tags integration` (space, not `=`). After changing migrations/queries run `moon run sqlc` and commit. Test doubles use **mockery** (Proto-pinned `mockery`, `.mockery.yaml` testify template, generated mock committed at `internal/todo/mocks`, drift-guarded by `root:mockery-check` — mirrors the sqlc generate/check pattern); after changing a mocked port run `moon run mockery` and commit. Repository doubles: the generated `mocks.Repository` for interaction/error-injection unit tests (`service_test.go`), a stateful in-memory fake `internal/todo/todotest` for end-to-end HTTP tests + no-DB app wiring. mockery release assets are `mockery_{version}_{Darwin|Linux}_{x86_64|arm64}.tar.gz` (note `x86_64`, not `amd64`).
- **PostgreSQL-only** (PR #9 `8a46286`, session 008). The in-memory `memory` adapter and the `--store` toggle were removed to cut boilerplate consumers would delete; `--database-url` is now **required** (`serve` fails fast without it). The `todo.Repository` port stays as the extension seam — add a datastore by implementing it (and wiring in `app.go`, or injecting via `app.WithRepository`). Postmortem `.journal/008/SUMMARY.md`.
- **Authorization tier is built** (PR #10 `13a1fe5`, session 005). Engine: **AWS Cedar via `cedar-go`**, committed as the sole engine (no portability layer) behind `internal/authz`. **Modular per-resource authz slices**: each domain ships `internal/<domain>/authz` (package `authz`, imported aliased — e.g. `todoauthz`, like the `todopostgres` alias) contributing `.cedar` policies + action UIDs (`Action::"<resource>:<verb>"`) + a repo-backed lazy fact resolver; the composition root merges all contributions into one `PolicySet` + a composite `EntityGetter`. Cross-cutting policies + shared principal types live in `internal/authz/base.cedar`; the principal entity type is reserved (a slice cannot shadow it). One **global deny-by-default** Huma middleware: every operation must declare `authz.Require(action[, idParam])` or `authz.Public()` on `Operation.Metadata` (undeclared → 403). CRITICAL: install the middleware BEFORE `huma.Register` (Huma freezes per-op middleware at register time) and stamp OpenAPI security AFTER (the Install/Finalize split). `Require(action, idParam)` binds the path param → `Resource = <Type>::"<id>"` (no load); attribute policies load entities lazily via the request-scoped getter (binds ctx at construction, captures first load error → 500). Decisions use the shared RFC 9457 `problem` writer: allow→next, deny→403 (401 anonymous), error→500. **Authentication is DEFERRED**: an `Authenticator` seam (`Authenticate(huma.Context) (Principal, error)`); the shipped impl is `internal/authz/apikey` (X-API-Key/Bearer → `APIKeyStore` port → postgres `api_keys` table) — a replaceable placeholder; swap for real JWT/OIDC. `app.WithAuthenticator` is the test/extension seam. Config: `--authz-enabled` (default true), `--authz-policy-dir`. Dev mock keys seed via `hack/sql/0002_seed_api_keys.sql` (dev-only, NEVER a migration, so they can't reach a real deploy); `omit_unused_structs: true` in `sqlc.yaml` keeps the shared `api_keys` migration from emitting an `ApiKey` model into the todo sqlc package. Design doc `.journal/005/AUTHZ_TIER.md`; container-backed tests in `internal/integration` (`apikey_store_test`, `authz_e2e_test`).
- **All previously-deferred feature seams are now BUILT** (session 011). The "future slices" list is empty; the code + README are the source of truth. Three tiers landed:
  - **API versioning** (PR #16 `a485f7e`): URI-path `/v1` via Huma's first-class `huma.NewGroup`. Applied in ONE place — `registerResources` in `app.go` (`v1 := huma.NewGroup(api, apiVersionV1); httpapi.Register(v1, service)`) — which is the single `Registrar` feeding BOTH the running router and the server-less spec export, so `/v1` stays consistent and `openapi-check` doesn't drift. A Huma group IS a `huma.API` (so adapters keep declaring unprefixed `Path: "/todos"` and the prefix is a composition concern) and inherits the parent API's middleware (so the root-API authz still applies to `/v1` routes — verified vs Huma v2.38.0 `group.go`). Infra routes (`/healthz`,`/readyz`,`/metrics`) stay unversioned. Adding `/v2`: a sibling `huma.NewGroup(api, "/v2")` — documented in README "API versioning". Adapter/authz UNIT tests deliberately stay at `/todos` (they test components in isolation on a bare API); composed tests use `/v1/todos`.
  - **Rate limiting** (PR #17 `867662f`): per-client (IP), on by default (`--rate-limit-enabled`, `--rate-limit-rps`=10, `--rate-limit-burst`=20). A Huma middleware installed BEFORE authz (`RouterDeps.InstallRateLimit`, runs outermost) so an over-limit request gets RFC 9457 429 + `Retry-After` before the credential store is hit; infra routes bypass Huma so they are exempt. In-process token bucket (`golang.org/x/time/rate`, idle-evicted per-key buckets, janitor stopped on shutdown) behind the `internal/ratelimit` `Limiter` port (the Redis/distributed seam). Pluggable `KeyFunc` — default `adapterhttp.ClientIPKeyFunc` (the spoof-safe resolved IP); swap for a principal-based key to limit authenticated callers. Ships `Retry-After` only; IETF `RateLimit` headers are a documented (draft) enhancement.
  - **OpenTelemetry tracing** (PR #18 `6625ab1`): opt-in (`--tracing-enabled`, default FALSE — needs a collector). OTLP/HTTP exporter configured ENTIRELY via standard `OTEL_*` env vars (no bespoke flags). otelhttp server spans at the edge (`RouterDeps.Tracing` wraps the mux, `traceableRequest` filters infra routes) named by operation via the `observability.TraceSpanNamer` Huma middleware; otelpgx DB spans (`postgres.Config.Tracing`). Bootstrap in `internal/observability/tracing.go` (`NewTracerProvider` returns a flush func or nil; W3C propagator; service.name/version default to app name + build version, overridable via `OTEL_SERVICE_NAME`/`OTEL_RESOURCE_ATTRIBUTES`); flushed on shutdown. GOTCHA: keep only `WithTelemetrySDK` schema-bearing in `resource.New` (WithAttributes/WithFromEnv schemaless) to avoid schema-URL merge conflicts; semconv pinned at `v1.39.0` (latest bundled by otel v1.44).
  - Keyset pagination was already built (session 010). Service-level manual tracing spans are intentionally NOT built (HTTP+DB scope).
- **CI runs the container-backed integration suite** (PR #11 `c9a6bbf`, session 009). The old "`.disabled` workflows need a Docker-capable runner" claim was false: `.github/workflows/ci.yml` runs `moon ci` on `ubuntu-latest` (Docker preinstalled). `test-integration` now carries `runInCI: true`; it is deliberately kept OUT of the `root:check` aggregate so a local `check` stays hermetic (no Docker). - **Independent security review (Codex) remediated** (session 010, three PRs). (1) **API keys are stored hashed** (PR #13 `ff55a2e`): the `api_keys` table column is `key_hash` (lowercase-hex SHA-256), the `apikey.Store` hashes the presented credential and looks up by digest; hashing lives BELOW the `APIKeyStore` port (Authenticator + mocks unchanged); the dev seed + integration fixtures hash in SQL via `encode(sha256($1::bytea),'hex')`. No constant-time compare (indexed equality on a preimage-resistant digest isn't a timing oracle). (2) **`GET /todos` is keyset-paginated** (PR #14 `879e2be`): `limit` (default `todo.DefaultPageSize=20`, max `MaxPageSize=100`; the max is a static Huma `maximum` tag AND the service clamps) + an opaque `cursor` over `(created_at,id)`, response carries `nextCursor`. The cursor is transport-only (`httpapi/cursor.go` base64url JSON); the port speaks `PageQuery`/`PageResult`; the +1 over-fetch lives in the adapters (postgres repo + `todotest` fake); a malformed cursor → `todo.ErrInvalidCursor` → 422. A `(created_at,id)` index was added to the todos migration. (3) **sqlc binary integrity-pinned** (PR #15 `f2c5210`): sqlc publishes no upstream checksum, so `.moon/proto/sqlc.sha256` pins the per-platform binary SHA-256 and the `moon` `sqlc-verify` task (dep of `sqlc`/`sqlc-check`, in `root:check`) verifies the resolved binary BEFORE running it (via `proto install`/`proto bin`, no execution; fail-closed). The other 3 Proto plugins already verify via `checksum-url`.
- GOTCHA: `root:format` is **`golangci-lint fmt`** (aligns struct tags WITHIN the backticks), NOT plain gofmt — `gofmt -w` is insufficient; run `proto run golangci-lint -- fmt --config .golangci.yml` (no `--diff`) to auto-fix. The stale-golangci-cache gotcha (session 007) still bites cross-worktree: `golangci-lint cache clean` before `root:check`. The Docker Compose stack seeds **3 demo todos** (ids `1111…`/`2222…`/`3333…`) in addition to the api_keys, so `/todos` is non-empty out of the box.
- GOTCHA (session 011): a trailing-slash `.gitignore` pattern matches only a real DIRECTORY, never a symlink. `.claude` in this checkout is a symlink → `.agents` (local convenience), so the old `.claude/` rule never ignored it; `.gitignore` now uses bare `.claude` (matches dir OR symlink) — keep it slashless. Local maintainer tooling is ignored: `.codex-security/` (scan output) and `.agents/skills/codex-security-scan/` (a local skill under the otherwise-committed `.agents/skills`). Supply-chain: only sqlc is repo-pinned (`.moon/proto/sqlc.sha256`) because it ships no checksums; golangci-lint/goose/mockery verify via upstream `checksum-url` and are deliberately NOT repo-pinned (documented in README "CI and Security").
- **Container image is built by the hand-rolled multi-stage `Dockerfile`** (distroless `static-debian12:nonroot` runtime, Go-native cross-compile via `--platform=$BUILDPLATFORM` + GOOS/GOARCH, build-args VERSION/COMMIT/DATE → ldflags). `release.yml`/`release-dry-run.yml` drive it via a buildx matrix (linux/amd64+arm64) + push-by-digest + `imagetools create` manifest + per-platform smoke tests; `compose.yaml` reuses it for the day-one stack. **Higher-level builders were assessed and NOT adopted (session 014, research-only, no change):** moon's `docker file/scaffold/setup/prune` layer (weakest — monorepo dep-install tool; default gen regresses the image), `ko` (best fit for this Go service — all runtime assets are `go:embed`'d so the image is just the binary, native `kos:` GoReleaser integration — but Go-only), and Cloud Native Buildpacks (only cross-language option + `pack rebase`/SBOM, but larger image, multi-arch friction, least transparent). Decision deferred; if revisiting, read `.journal/014/SUMMARY.md` first (it carries the tradeoffs + blast radius). NOTE: GoReleaser had a `buildpacks` block in 0.179.0 (2021) but removed it in 2022 — today buildpacks means driving `pack` directly.
